# Smart-Cat 开发日志

本文档用于记录 Smart-Cat 项目从想法验证到系统闭环的主要开发过程。项目当前定位为“基于视觉与多传感器融合的边缘智能称重识别一体机及 Web 管理平台”，核心目标是完成从边缘设备采集、AI 识别、称重计价、语音播报、MQTT 上报到 Web 后台管理的完整工程闭环。

## 当前项目状态

截至 2026-06-29，项目已经完成以下主链路：

```text
摄像头拍照
-> RK3576 NPU / RKNN 商品识别
-> HX711 称重
-> 商品表计价
-> SYN6288 语音播报
-> 本地交易记录
-> MQTT 上报
-> Web 后台同步与管理
```

当前系统已经具备边缘设备端、通信层和 Web 管理平台三部分，初步形成工业软件项目雏形。

## 阶段一：项目定位与方案设计

### 阶段目标

最初项目设想为“智能商品识别系统”，后续逐步升级为：

```text
基于视觉与多传感器融合的边缘智能称重识别一体机
```

项目不再只做摄像头识别，而是构建“视觉为主、称重校验、语音补盲、平台管理”的多模态系统。

### 核心思路

```text
视觉识别：判断商品类别
称重模块：读取重量并参与计价
语音模块：播报结果，降低用户看屏幕成本
MQTT：将交易数据上传到平台
Web 后台：管理交易、商品、异常、统计
```

### 阶段成果

- 明确项目技术路线：嵌入式 + 边缘 AI + 传感器 + MQTT + Web 后台。
- 明确论文/报告结构：硬件传感子系统、边缘 AI 子系统、云/平台数据子系统。
- 初步形成商业计划书方向：面向生鲜零售、自助称重、无人值守收银。

## 阶段六：HX711 称重模块接入

### 阶段目标

接入 HX711 称重模块，读取真实重量，并与识别结果一起完成计价。

### 当前接线

```text
DT/DOUT: GPIO4_A6 -> GPIO number 134
SCK/CLK: GPIO4_A4 -> GPIO number 132
VCC: 3.3V
GND: GND
```

### 已完成内容

- 编写 HX711 读取脚本。
- 支持 GPIO 后端切换：

```text
sysfs
gpiod
```

- 支持原始值读取、去皮、校准、滤波。
- 使用约 550g 已知重量完成校准。
- 最终读数接近真实重量：

```text
Raw filtered: 292608.00
Weight: 549.95 g
```

### 遇到的问题与处理

| 问题 | 现象 | 原因分析 | 处理方式 |
| --- | --- | --- | --- |
| 原始值一直不变 | 多次读取都是固定大数 | GPIO 读取时序或引脚访问方式不稳定 | 改进 GPIO 访问方式 |
| 读数出现 `-1` 或 `8388607` | 偶发极端值 | HX711 时序干扰、毛刺、异常采样 | 加入滤波，使用 filtered raw 而不是简单均值 |
| 校准后重量偏差很大 | 均值被异常值污染 | 极端值参与平均 | 使用离群值过滤后的 raw filtered |

### 阶段成果

称重模块从“能读数”升级为“可用于计价的稳定重量输入”，完成多传感器融合的关键一环。

## 阶段八：SYN6288 动态语音播报

### 阶段目标

接入 SYN6288 TTS 模块，实现动态报价语音播报。

### 当前接线

```text
鲁班猫 引脚8  / UART_TX -> SYN6288 RXD
鲁班猫 引脚10 / UART_RX -> SYN6288 TXD
鲁班猫 5V                 -> SYN6288 VCC
鲁班猫 GND                -> SYN6288 GND
```

### 已完成内容

- 启用 UART10，对应设备：

```text
/dev/ttyS10
```

- 编写 SYN6288 串口协议脚本。
- 参考 STM32 示例代码修正协议帧。
- 支持音量、语速、背景音等控制参数。
- 成功接入完整流程：

```bash
./bin/run_smart_scale.sh --tts syn6288 --tts-port /dev/ttyS10 --tts-volume 3
```

### 遇到的问题与处理

| 问题 | 原因分析 | 处理方式 |
| --- | --- | --- |
| 最初没有声音 | UART10 未启用、协议帧与模块要求不完全一致 | 启用 `uart10-m1`，参考 STM32 代码修正帧格式 |
| 声音过大 | 默认音量偏高 | 增加 `--tts-volume` 参数 |
| 是否使用 US516P6 | 离线命令词模块动态播报能力有限 | 改用 SYN6288 动态 TTS |

### 阶段成果

实现“识别 + 称重 + 计价 + 语音报价”，增强了交互体验，也为后续“语音补盲”保留接口。

## 阶段九：MQTT 数据上报与网络问题处理

### 阶段目标

让板子把每笔交易上报到 MQTT broker，实现设备端与平台端解耦。

### 已完成内容

- 编写 MQTT 发布脚本：

```text
scripts/mqtt_publisher.py
```

- 完整流程默认启用 MQTT：

```text

```

- 交易成功后终端显示：

```text

```

- MQTTX 成功接收到交易记录。

### 遇到的问题与处理

| 问题 | 现象 | 原因分析 | 处理方式 |
| --- | --- | --- | --- |
| DNS 解析失败 | `Temporary failure in name resolution` | 板子网络不可达或无 DNS | 改用无线网卡连接 WiFi |
| USB 网络能 SSH 但不能上网 | 共享网络和路由不完整 | USB RNDIS 不适合作为稳定公网链路 | 使用 USB WiFi |
| MQTTX 收不到消息 | 板子发布 topic 与订阅 topic 不一致 | `-` 和 `_` 差异导致主题不同 | 统一 topic |
| Web 后台连不上 MQTT | 默认连接 `127.0.0.1:1883`，本机无 broker | 缺少电脑端 `config/mqtt.json` | 配置与板子相同的 broker 和 topic |

### 阶段成果

设备端交易数据可以实时进入 MQTT broker，具备工业设备数据上报能力。



### 设计取舍

实时交易闭环不直接依赖大模型，而采用边缘 AI + 规则逻辑完成，原因是：

```text
降低网络依赖
减少延迟
避免大模型输出不确定
更符合工业现场稳定性要求
突出边缘智能设备价值
```

大模型更适合作为后续平台智能助手，用于经营分析、异常总结和自然语言查询，而不是直接放进称重收银实时控制链路。


这一步让项目从“命令行演示流程”进一步接近真实边缘设备：设备上电后可以一直工作，用户放上商品后自动产出交易；同时仍保留 `bin/run_smart_scale.sh` 作为单次调试入口，便于排查摄像头、识别、语音、MQTT 等单个问题。

### 常驻服务空秤同步优化

在初版常驻服务中，设备是在重量稳定并准备交易前同步远程参数和设备策略。这样虽然能保证每次交易使用最新配置，但会把 MQTT 拉取耗时放到用户等待结算的关键路径上。

本次调整后，常驻服务改为只在 `IDLE` 空秤状态下同步配置：

```text
空秤待机
-> 按 sync-interval 定期同步参数和策略
-> 检测到商品放上秤
-> 等待读数稳定
-> 直接使用本地最新配置执行交易
-> 等待商品拿走
-> 回到空秤后继续同步
```

新增参数：

```text
--sync-interval 15   空秤时远程同步间隔，单位秒
```

这样设计的好处是：空秤时允许云端/平台改变边缘设备行为；称重交易过程中保持本次业务逻辑稳定，不因为网络波动影响识别和报价响应速度。若 Web 后台刚保存策略时秤上已经有商品，本次交易可能仍使用上一次空秤同步到本地的策略，商品拿走后下一轮空秤同步会更新到最新策略。

### 设备心跳与服务状态上报

为了给后续语音补盲和远程运维打底，常驻服务新增设备心跳和服务状态事件。

新增参数：

```text
--heartbeat-interval 60   常驻服务心跳间隔，单位秒；设置为 0 可关闭
--events records/device_events.jsonl   板端本地设备事件记录文件
--status-file records/service_status.json   当前服务状态快照
```

新增事件类型：

```text
service_heartbeat  常驻服务心跳，包含当前状态、当前重量、交易次数
service_state      服务状态变化，例如 IDLE、WAIT_STABLE、RUN_TRANSACTION、WAIT_REMOVE
```

事件会写入板端 `records/device_events.jsonl`，并通过 MQTT `device-events` topic 上报到 Web 后台。Web 后台的设备状态接口现在优先根据 `service_heartbeat` / `service_state` 判断设备在线状态，并展示服务状态和当前重量。这样即使长时间没有交易，后台也能知道设备仍在运行。

商品修正逻辑复用 Web 后台已有的人工修正能力：默认选择最近一笔交易，更新商品类别和单价，重新计算总价，并把交易状态改为 `manually_confirmed`，页面展示为“已确认”。同时在交易记录 `corrections` 中保存修正前后的信息，修正来源标记为 `voice_mobile`。这样更符合“刚识别完马上用手机语音纠正”的现场演示方式。

板端 `voice_command_mqtt.py` 新增 `speak_text` 命令。Web 后台完成商品修正后，不再要求板端读取电脑端修正后的交易记录，而是直接通过 MQTT 下发修正后的播报文本，由板端 SYN6288 立即播报。这样解决了“Web 后台修正记录”和“板端本地记录”不完全一致时的语音播报问题。

典型演示语句：

```text
这是苹果
改成番茄
不是香蕉，是橙子
播报价格
当前重量
```

## 2026-07-06：常驻服务摄像头复用尝试与回退

常驻称重触发服务此前已经复用 RKNN runtime，但每次触发交易时仍会重新打开摄像头、设置分辨率、预热、拍照并释放。为了缩短“放上商品到语音报价”的等待时间，本次曾尝试新增 `CameraCapture` 复用层：

```text
smart_scale_demo.py   新增 CameraCapture，单次模式仍然拍完即释放
smart_scale_service.py 常驻模式曾持有 CameraCapture，设备号和分辨率不变时复用
```

当前行为：

```text
./bin/run_smart_scale.sh
  单次模式，每次运行都会重新打开摄像头和初始化 RKNN

./bin/run_smart_scale_service.sh
  常驻模式，第一次交易初始化 RKNN，后续交易复用 RKNN；摄像头最终改回每笔交易重新打开
```

使用 `--subprocess-transaction` 时仍然回到子进程隔离模式，每笔交易会重新初始化资源。

复用摄像头后发现一个问题：USB 摄像头和 OpenCV 后端可能保留旧帧缓冲，导致物体已经更换，但交易识别仍拿到上一笔画面，从而出现未训练物体也被高置信度识别为苹果、香蕉或橙子的现象。虽然 `CameraCapture` 中加入了缓冲刷新逻辑，但实测仍不够稳定，因此常驻服务最终回退为每笔交易重新打开摄像头、拍照并释放。当时 RKNN runtime 仍然复用，保留主要耗时优化；摄像头部分优先保证画面可靠。

随后继续排查发现：常驻服务第一笔交易识别正常，但后续交易仍可能出现置信度异常偏高。为排除 RKNN runtime 长时间跨交易复用带来的状态问题，服务逻辑进一步改为“空秤预加载、单笔使用、交易后释放”：设备空闲时提前初始化 RKNN predictor，用户放上商品后直接使用预加载实例推理，交易完成后立即释放，下一轮回到空秤状态再重新预加载。这样既避免在用户等待交易时加载模型，又不让同一个 RKNN runtime 连续跨多笔交易复用。

常驻服务稳定后，终端输出也做了收敛。默认模式会静默 MQTT retained 同步脚本和 RKNN Lite 初始化时的大段底层日志，只保留类似 `[IDLE] ready: sync=ok, rknn=ready, weight=0.2 g` 的摘要。需要排查 MQTT topic、保留配置或 RKNN runtime 细节时，可使用 `./bin/run_smart_scale_service.sh --verbose` 恢复详细输出。

在确认异常主要来自 RKNN runtime 跨交易复用后，摄像头重新改为常驻打开。最终常驻服务采用的资源策略是：摄像头在空秤准备阶段打开并保持，交易拍照前主动刷新缓冲帧，避免旧帧；RKNN 在空秤阶段预加载，单笔交易使用后立即释放，下一轮空秤再重新加载。这样摄像头初始化耗时被消除，同时避免 RKNN 长时间跨多笔交易复用带来的置信度异常。

## 2026-07-13：轻量商品记忆库与“教一次，下次记住”

在答辩后继续验证“模型没有训练过的物体，能否上传或确认一次后下次识别”的方向。相比直接上 YOLOE / YOLO-World 这类开放词表检测模型，本项目先采用更适合板端落地的轻量商品记忆库：

```text
未知 / 待确认交易
-> Web 后台或手机语音确认商品
-> Web 通过 MQTT 下发 bind_memory
-> 板端读取本地交易图片
-> 提取轻量图像特征
-> 写入 records/product_memory.jsonl
-> 下次 YOLO/RKNN 无法可靠识别时，先查商品记忆库
-> 相似度足够高则 status = memory_matched 并自动计价
```

这次实现没有让电脑 Web 后台读取板端图片。前一版曾尝试在 Web 后台确认交易时直接从 `source_image` 读取图片并生成记忆，但实际运行时 Web 后台在电脑上，交易图片在板子上，电脑端路径会变成：

```text
D:\vscode\Smart-Cat\outputs\transactions\<transaction_id>.jpg
```

而真实图片位于板端：

```text
~/Smart-Cat/outputs/transactions/<transaction_id>.jpg
```

因此最终改为方案 B：Web 后台只负责确认“这是什么商品”，真正的图片读取、特征提取和记忆库写入都在板端完成。这更符合云边协同架构：平台下发业务指令，边缘设备使用本地数据改变后续识别行为。

### 关键改动

```text
scripts/product_memory.py
  新增轻量特征提取、JSONL 记忆保存、相似度检索

scripts/voice_command_mqtt.py
  新增 bind_memory 命令，板端收到后生成 records/product_memory.jsonl

scripts/recognize_product_detector_rknn.py
  unknown / low_confidence / needs_confirm 时自动查询商品记忆库

scripts/web_dashboard.py
  Web 后台确认商品后下发 bind_memory，不再依赖电脑访问板端图片

web/static/app.js
  交易详情显示识别来源、商品记忆状态、相似度和记忆 ID
```

### 轻量特征方案

第一版没有引入新的深度特征模型，而是使用：

```text
2x2 HSV 颜色直方图
+ 边缘密度
+ L2 归一化
+ 余弦相似度
```

这样做的原因是实现成本低、板端可直接运行、无需重新训练模型，适合先验证“商品记忆”业务闭环。后续如果要增强光照、角度和背景变化鲁棒性，可以替换为 MobileNet / EfficientNet-lite / RKNN 特征提取模型。

### 实测结果

板端实测流程：

```text
1. 放置未训练商品，产生 unknown / needs_confirm 交易
2. 在 Web 后台交易详情中选择正确商品并确认修正
3. Web 下发 bind_memory
4. 板端生成 records/product_memory.jsonl
5. 再次放置相似商品
6. 系统进入 memory_matched，并自动计价
```

实测商品记忆相似度可达到约 `0.8`，说明即使不重新训练 YOLO 模型，也能通过“人工确认 + 板端特征库”让系统具备一定的少样本记忆能力。这是项目从“识别三类固定商品”升级到“可被现场逐步教会”的关键亮点。
