## 第 5 页：方案选型综述

标题：

```text
方案选型：围绕实时性、可部署性和业务可靠性
```

页面内容表格：

| 设计点 | 可选方案 | 本项目选择 | 主要原因 |
| --- | --- | --- | --- |
| 推理位置 | 云端 / PC / 板端 | 板端 NPU | 低延迟，弱网络可运行 |
| 模型形态 | 分类 / 检测 | YOLO11 检测 | 可处理无框、未知和低置信度 |
| 部署格式 | PyTorch / ONNX / RKNN | RKNN | 适配 RK3576 NPU |
| 通信方式 | HTTP / WebSocket / MQTT | MQTT | 适合轻量设备和主题通信 |
| 数据存储 | SQLite / JSONL | JSONL | 轻量、易调试，适合原型阶段 |

演讲提示：

```text
这些选择不是为了堆技术，而是围绕称重收银设备的实时性、部署成本和可靠性来决定。
```

## 第 6 页：边缘 AI 与 RKNN/NPU 对比

标题：

```text
为什么使用 RKNN/NPU 边缘推理
```

页面内容：

```text
训练：YOLO11
交换：ONNX
部署：RKNN
运行：RKNN Lite + RK3576 NPU
```

对比表：

| 方案 | 优点 | 不足 |
| --- | --- | --- |
| 云端识别 | 算力强，模型更新方便 | 依赖网络，延迟不稳定 |
| PC GPU | 推理速度快 | 不适合称重终端部署 |
| 板端 CPU | 部署简单 | 性能不足 |
| RK3576 NPU | 本地实时，功耗较低 | 需要模型转换和适配 |

演讲提示：

```text
称重收银需要现场快速反馈，所以最终选择把模型转成 RKNN，在鲁班猫 3 的 NPU 上运行。
```

## 第 7 页：识别业务设计对比

标题：

```text
不是所有识别结果都直接结算
```

页面内容：

```text
自动结算条件：
Top1 confidence >= 0.75
Top1 - Top2 confidence gap >= 0.15

不满足条件：
进入低置信度 / 待确认 / 未知商品
不计算总价
等待 Web 后台或语音补盲确认
```

建议配图：

```text
检测框 -> 置信度判断 -> 自动结算 / 待确认分支流程图
```

演讲提示：

```text
真实业务中错误结算比拒绝结算更严重，因此我加入了置信度阈值和 gap 判断，把不确定结果交给人工兜底。
```

## 第 8 页：MQTT 云边协同设计

标题：

```text
相同模型 + 不同策略 = 不同业务效果
```

页面内容：

| 数据类型 | MQTT 类型 | 作用 |
| --- | --- | --- |
| 交易记录 | 普通消息 | 每笔交易上报后台 |
| 设备事件/心跳 | 普通消息 | 记录运行状态 |
| 语音命令 | 普通消息 | 即时执行，避免重复 |
| 运行参数 | Retained | 新上线设备立即获取最新配置 |
| 设备策略 | Retained | 表示当前业务规则 |

补充示例：

```text
同一个检测模型：
宽松策略 -> 可直接结算
保守策略 -> 进入待确认
促销策略 -> 按折扣规则计价
```

演讲提示：

```text
后台可以通过 MQTT retained 消息远程改变板端行为，设备不需要频繁改代码。
```
