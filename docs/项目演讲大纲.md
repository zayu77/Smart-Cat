# Smart-Cat 答辩演讲大纲

## 演讲定位

本次答辩建议围绕老师要求的三个部分展开：

```text
一、方案架构及实际应用领域：简要说明系统做什么、用在哪里
二、方案选型（设计）的主要考虑和综述对比：说明为什么这样设计
三、进度及成果：重点展示已经完成的功能、闭环和可演示成果
```

推荐演讲时间：8 - 10 分钟。第三部分“进度及成果”建议占 50% 以上时间。

## 一、方案架构及实际应用领域（简要）

### 1. 项目要解决的问题

Smart-Cat 面向生鲜、水果、散装商品等称重收银场景。传统流程需要人工识别商品、查找单价、称重、计价和播报，存在效率低、易出错、数据不集中等问题。

本项目希望实现一套“边缘智能称重识别一体机”原型，让设备在本地自动完成：

```text
商品识别
重量读取
价格计算
语音播报
交易记录
后台同步
人工确认和语音补盲
```

### 2. 总体架构

系统由边缘设备、MQTT 通信和 Web 后台三部分组成：

```text
感知层：
USB 摄像头、HX711 称重模块、称重传感器、SYN6288 语音模块

边缘计算层：
鲁班猫 3 / RK3576，负责称重触发、拍照、RKNN/NPU 推理、计价和播报

通信协同层：
MQTT 负责交易上报、设备事件、运行参数、策略下发和语音命令

平台管理层：
电脑 Web 后台负责交易流水、商品档案、统计分析、设备状态、运行参数和策略管理
```

可以用一句话概括：

```text
边缘端负责实时交易，Web 后台负责管理配置，MQTT 负责云边协同。
```

### 3. 实际应用领域

本项目可以对应以下应用方向：

```text
水果店、生鲜摊位、超市散装商品称重收银
校园/实验室智能零售终端原型
边缘 AI 工业软件教学示例
低成本智能称重设备改造
面向视障顾客或无屏场景的语音辅助收银
```

这一部分答辩时简要讲清楚即可，不需要展开太久。

## 二、方案选型（设计）的主要考虑和综述对比

### 1. 为什么采用边缘推理，而不是云端识别

对比：

| 方案 | 优点 | 不足 | 本项目选择 |
| --- | --- | --- | --- |
| 云端识别 | 算力强，模型更新方便 | 依赖网络，延迟不稳定，隐私和部署成本更高 | 不作为主流程 |
| PC 端识别 | 调试方便，性能较强 | 不适合嵌入到称重终端 | 用于训练和开发 |
| 板端 NPU 推理 | 本地实时、低延迟、弱网络可运行 | 需要模型转换和适配 | 作为最终部署方案 |

本项目选择 RK3576 NPU，是因为称重收银需要现场实时反馈，不能每笔交易都依赖网络请求。

### 2. 为什么从分类模型转向检测模型

早期分类模型可以判断图片属于哪一类商品，但在空画面、未知物体或干扰背景下，容易强行给出一个类别。

检测模型的优势：

```text
可以判断有没有检测框
可以过滤无商品画面
可以结合置信度进入待确认
更接近真实商品识别场景
```

当前识别规则：

```text
Top1 confidence >= 0.75
Top1 - Top2 confidence gap >= 0.15
满足条件才自动结算
否则进入低置信度 / 待确认 / 未知商品
```

这体现了项目不是“模型一定给答案”，而是把不确定结果交给人工确认，降低错误结算风险。

### 3. 为什么选择 RKNN 部署路线

模型部署路线：

```text
YOLO11 训练
-> 导出 ONNX
-> 转换 RKNN
-> RKNN Lite 调用 RK3576 NPU
```

对比：

| 方式 | 适用场景 | 评价 |
| --- | --- | --- |
| PyTorch | 训练和验证 | 不适合直接部署到板端 |
| ONNX | 中间交换格式 | 便于转换，但不是最终运行形态 |
| RKNN | Rockchip NPU 部署 | 适合鲁班猫 3 的边缘推理 |

### 4. 为什么用 MQTT 做云边协同

MQTT 适合轻量级设备通信，主题清晰，便于电脑后台和板端分离部署。

本项目中 MQTT 的分工：

| 数据类型 | MQTT 类型 | 设计原因 |
| --- | --- | --- |
| 交易记录 | 普通消息 | 每笔交易是一次性事件 |
| 设备事件/心跳 | 普通消息 | 用于记录过程状态 |
| 语音命令 | 普通消息 | 避免设备重连后重复执行旧命令 |
| 运行参数 | Retained 消息 | 设备上线后要立即拿到最新配置 |
| 设备策略 | Retained 消息 | 表示当前应执行的业务规则 |

可以重点说明：

```text
相同模型 + 不同策略 = 不同业务效果
```

模型负责判断画面中像什么商品；策略负责决定是否自动结算、是否待确认、是否促销计价。后台可以远程改变设备行为，而不必频繁修改板端代码。

### 5. 为什么保留人工确认和语音补盲

真实业务中不能完全依赖模型输出。低置信度或未知商品进入待确认后，可以通过 Web 后台或手机语音输入进行修正。

示例流程：

```text
设备识别不确定
-> 不计算总价
-> 语音提示需要人工确认
-> 手机输入“这是苹果”
-> 后台修改商品类别
-> 重新计算总价
-> 记录修正前后信息
-> 板端播报修正结果
```

这让系统具备“AI 自动处理 + 人工兜底”的实际可用性。

## 三、进度及成果（重点）

### 1. 已完成硬件闭环

当前已经完成以下硬件接入：

```text
鲁班猫 3 / RK3576 主控
USB 摄像头拍照
HX711 称重模块读取重量
称重传感器校准
SYN6288 串口语音播报
无线网络连接 MQTT Broker
```

板端可以完成从放置商品到播报价格的一整套流程。

### 2. 已完成边缘 AI 识别闭环

已经完成：

```text
整理 YOLO11 检测数据集
训练 apple / banana / orange 检测模型
导出 ONNX
转换 RKNN
在 RK3576 NPU 上运行检测推理
保存检测框预览图 outputs/detections/交易ID.jpg
```

识别结果不再只输出类别，而是结合检测框、置信度和 Top1/Top2 差值生成业务状态。

### 3. 已完成常驻称重触发服务

项目从“每次运行一次脚本”升级为“常驻服务”：

```text
空秤待机
空闲时同步远程参数和策略
空闲时预加载 RKNN
保持摄像头打开并刷新缓冲帧
检测到重量超过阈值
等待重量稳定
自动拍照识别
完成计价、播报、记录和上报
等待商品移走后进入下一轮
```

这个成果很重要，因为它让项目更接近真实设备，而不是一次性 Demo。

### 4. 已完成交易业务闭环

一笔交易记录包含：

```text
交易 ID
时间
设备 ID
商品 ID / 商品名称
识别状态
置信度和置信度差值
重量
单价
总价
语音播报文本
策略版本和计价模式
修正前后信息
```

对于可靠识别结果，系统自动计价；对于低置信度、未知商品或 gap 不足的情况，系统不直接计算总价，等待人工确认。

### 5. 已完成 Web 后台

Web 后台已经实现：

```text
监控概览：交易数、销售额、总重量、待确认数量
交易流水：按时间、商品、状态筛选，详情查看，CSV 导出，清空流水
商品档案：商品 ID、名称、单价、单位、启用状态、备注、价格修改记录
统计分析：销售趋势、销量排行、识别质量
运行参数：置信度阈值、语音音量、MQTT 上报开关
设备策略：低置信度动作、计价模式、语音模板、策略回滚
设备事件：心跳、服务状态、策略应用、语音命令记录
语音补盲：手机/Web 输入自然语言修正交易
```

答辩时建议重点展示 Web 后台，因为它最直观地体现“工业软件平台”的完整性。

### 6. 已完成云边协同能力

当前已实现：

```text
板端交易记录通过 MQTT 上报
Web 后台订阅交易 topic 并写入本地记录
Web 后台下发运行参数 retained 消息
Web 后台下发设备策略 retained 消息
板端空秤时自动拉取最新参数和策略
板端回传设备事件和心跳
Web/手机端发送语音命令
板端订阅语音命令并执行 SYN6288 播报
```

这一部分可以作为项目技术亮点：电脑 Web 后台和板端是分开的，后台能够远程改变设备行为。

### 7. 已完成语音补盲和无障碍查询

语音功能包括两类：

| 类型 | 示例 | 作用 |
| --- | --- | --- |
| 查询命令 | “重量”“价格”“最近交易”“状态” | 不看屏幕也能获取信息 |
| 纠错命令 | “这是苹果”“改成香蕉” | 对待确认交易进行人工修正 |

其中纠错命令会真正修改交易流水，重新计价，并记录修正信息。

### 8. 演示建议

答辩现场建议按这个顺序演示：

```text
1. 打开 Web 后台，展示监控概览
2. 启动板端常驻称重服务
3. 放上苹果/香蕉/橙子，展示自动识别、称重、计价、播报
4. Web 后台刷新交易流水
5. 打开交易详情，展示置信度、策略版本、计价明细
6. 放一个未知或不确定物体，展示待确认状态
7. 手机输入“这是苹果”，展示语音补盲修正
8. 后台调整运行参数或策略，说明板端空秤时拉取并应用
9. 展示设备事件日志和策略应用记录
```

## 结尾总结

可以这样收束：

```text
Smart-Cat 不是一个单独的识别模型 Demo，而是一套边缘智能称重识别系统。
它把摄像头、称重传感器、NPU 推理、语音播报、MQTT 云边协同和 Web 后台管理连接成了完整闭环。
项目的重点在于：AI 自动处理、人工确认兜底、平台远程配置、设备可持续运行。
```
