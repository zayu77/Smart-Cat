# Smart-Cat 项目说明

## 项目定位

Smart-Cat 是一套基于鲁班猫 3 / RK3576 的边缘智能称重识别一体机。项目面向水果店、生鲜摊位、食堂窗口等小型零售场景，目标是把传统“人工识别商品、人工称重、人工算价”的流程改造成自动化闭环：用户将商品放上称重台后，设备自动完成图像采集、商品识别、重量读取、价格计算、语音播报、交易记录保存和 MQTT 数据上报。

项目不是单纯的图像识别演示，而是围绕“边缘设备 + 传感器 + AI 推理 + 语音交互 + Web 管理后台 + 云边协同”构建的一套完整原型系统。边缘端负责实时交易闭环，电脑端 Web 后台负责配置下发、商品维护、交易查看、人工确认和数据统计。

## 主要功能

当前系统已经形成以下功能闭环：

```text
称重触发：设备常驻运行，空秤待机，检测到物体放上秤并稳定后自动触发交易
视觉识别：USB 摄像头拍照，YOLO11 检测模型识别 apple / banana / orange
边缘推理：ONNX 模型转换为 RKNN，在 RK3576 NPU 上运行
称重计价：HX711 读取重量，结合商品单价计算总价
语音播报：SYN6288 通过 UART 播报商品、重量、单价和总价
数据上报：交易记录通过 MQTT 上报，电脑 Web 后台可实时同步
Web 后台：交易流水、筛选、详情、CSV 导出、商品维护、统计分析
人工确认：低置信度、未知商品可由后台或手机语音补盲修正
云边协同：Web 后台通过 MQTT retained 下发运行参数和设备策略
策略管理：支持策略版本、低置信度动作、计价模式、语音模板和回滚
```

对于模型未训练到的商品，系统不会强行把完整流程做成“假自动化”，而是进入待确认状态，并通过 Web 后台或手机语音输入完成补盲修正，例如“这是苹果”“改成番茄”。这使系统既保留自动识别效率，也保留人工兜底能力。

## 系统架构

系统整体分为四层：

```text
感知层：
USB 摄像头、HX711 称重模块、称重传感器、SYN6288 语音模块

边缘计算层：
鲁班猫 3 / RK3576，负责摄像头采集、NPU 推理、称重读取、计价、播报和本地记录

通信协同层：
MQTT topic 负责交易上报、运行参数下发、设备策略下发、设备事件回传和语音补盲命令

平台管理层：
电脑 Web 后台负责交易管理、商品档案、设备状态、统计分析、策略管理和人工修正
```

常驻服务的运行流程如下：

```text
空秤待机
-> 同步 Web 后台下发的运行参数和设备策略
-> 预加载 RKNN runtime，保持摄像头打开
-> 重量超过触发阈值
-> 多次采样确认重量稳定
-> 刷新摄像头缓冲帧并拍照
-> NPU 推理识别商品
-> HX711 稳定重量参与计价
-> SYN6288 播报报价
-> 写入交易记录并通过 MQTT 上报
-> 释放本次 RKNN runtime
-> 等待商品拿走，回到空秤待机
```

这个流程体现了项目从“运行一次脚本完成一次演示”升级为“设备上电后常驻运行”的工程化思路。

## 技术路线

项目从模型训练到边缘部署的技术路线为：

```text
数据集整理
-> YOLO11 检测模型训练
-> 导出 ONNX
-> 转换 RKNN
-> 鲁班猫 3 / RK3576 NPU 推理
-> 交易业务封装
-> MQTT / Web 后台联动
```

早期项目曾尝试分类模型路线，但分类模型在没有商品或出现未知物体时容易强行给出某一类结果。当前主线切换为 YOLO11 检测模型：有检测框且置信度达到阈值时才自动计价；无框或低置信度时进入待确认流程。这种方式更符合真实收银场景，也更容易在演示中说明系统的可靠性边界。

## 为什么采用 RKNN / NPU 边缘推理

本项目部署在鲁班猫 3 / RK3576 上。RK3576 属于 Rockchip ARM SoC，板端内置 NPU。普通 PyTorch 或 ONNX 模型虽然适合训练、调试和跨平台交换，但在板端直接运行通常无法充分调用 Rockchip NPU，容易退化为 CPU 推理，导致延迟高、功耗大、实时性差。

RKNN 是 Rockchip 面向自家芯片的神经网络模型格式和推理部署方案。将 YOLO11 模型从 ONNX 转换为 RKNN 后，可以使用 RKNN Lite 在板端调用 NPU 推理，更适合本项目这种“商品放上秤后要尽快报价”的边缘实时场景。

参考对比如下，具体速度会受模型大小、输入分辨率、量化方式、摄像头采集和后处理影响，表中数值用于说明技术选型差异：

| 硬件环境 | 推理方式 | 参考延迟 |
| --- | --- | --- |
| 普通 PC，无独立 GPU | CPU 跑 PyTorch / ONNX | 约 500 - 1000ms / 张 |
| PC + RTX 3090 显卡 | GPU 推理 | 约 10ms / 张 |
| 树莓派 4B | CPU 推理 | 约 3000ms / 张 |
| 鲁班猫 3 / RK3576 | RKNN 调用 NPU | 约 30 - 80ms / 张 |

因此，项目选择 RKNN 并不是简单地更换模型格式，而是为了让 AI 推理真正落到边缘 NPU 上。这个选择直接服务于项目目标：设备无需依赖云端识别，也不需要高功耗显卡，就能在本地完成商品识别和报价。

## 云边协同设计

项目吸收了“云边协同、远程配置、策略下发、版本回滚”的思想，但没有为了复杂而完整实现 WASM 沙箱运行时。当前采用更贴合项目规模的轻量方案：

```text
Web 后台保存运行参数
-> MQTT retained 下发 runtime-config
-> 板端空秤待机时拉取并应用

Web 后台保存设备策略
-> MQTT retained 下发 device-policy
-> 板端交易时按策略处理低置信度、未知商品和语音模板

板端交易、心跳、策略应用结果
-> MQTT device-events 回传
-> Web 后台可视化展示设备状态和策略应用记录
```

这种设计的意义在于：边缘设备不需要频繁改代码，后台就能调整设备行为。例如修改置信度阈值、语音音量、低置信度处理方式或计价策略后，板端会在空秤状态下同步配置，下一笔交易自动按新策略执行。

### 相同模型 + 不同策略 = 不同业务效果

云边协同并不一定意味着每次都要远程更新 AI 模型。对于本项目来说，YOLO11 + RKNN 模型负责回答“画面里像什么商品”，而设备策略负责决定“这个识别结果在业务上怎么处理”。因此，即使边缘端使用的是同一个 RKNN 模型，只要 Web 后台下发不同策略，最终业务效果也会不同。

例如同一笔识别结果：

```text
Top1: apple 0.72
Top2: orange 0.61
重量: 500g
```

如果后台策略比较宽松：

```text
置信度阈值: 0.70
低置信度动作: accept
计价模式: standard
```

设备可能会直接按苹果计价并播报报价。

如果后台策略更保守：

```text
置信度阈值: 0.75
Top1/Top2 差值阈值: 0.15
低置信度动作: needs_confirm
计价模式: standard
```

同样的识别结果会进入待确认，不计算总价，也不会直接播报“苹果多少钱”，而是提示人工确认。

如果后台切换促销策略：

```text
置信度阈值: 0.75
低置信度动作: needs_confirm
计价模式: discount_10_over_1000g
```

模型仍然不变，但当识别结果可靠且重量超过规则条件时，设备会按促销计价方式重新计算总价。

这说明项目中的“平台改变设备行为”并不是简单改一个参数显示值，而是通过 MQTT retained 策略下发，让边缘设备在不重新烧录、不重新训练模型的情况下改变业务规则。答辩时可以将其概括为：

```text
模型解决识别问题，策略解决业务决策问题。
同一个模型，在不同策略下可以表现为自动结算、待人工确认、暂停结算或促销计价。
```

## MQTT 普通消息与 Retained 消息

项目中同时使用了 MQTT 普通消息和 retained 消息。两者的区别可以理解为：

```text
普通消息：
只在发布那一刻发送给当前在线并订阅该 topic 的客户端。
如果客户端当时不在线，之后再上线通常收不到这条历史消息。

Retained 消息：
Broker 会把该 topic 的最后一条 retained 消息保存下来。
新客户端订阅该 topic 后，会立即收到这条“最新状态”。
```

在 Smart-Cat 中，两类消息承担不同职责：

| 数据类型 | Topic 示例 | 是否 Retained | 原因 |
| --- | --- | --- | --- |
| 交易记录 | `smart-cat/lubancat3-demo-001/transactions` | 否 | 每笔交易是一次事件，Web 后台在线订阅即可写入流水 |
| 设备事件/心跳 | `smart-cat/lubancat3-demo-001/device-events` | 否 | 心跳和事件是过程记录，不需要覆盖成最后一条 |
| 运行参数 | `smart-cat/lubancat3-demo-001/runtime-config` | 是 | 板端重启或空秤待机时，需要拿到后台保存的最新参数 |
| 设备策略 | `smart-cat/lubancat3-demo-001/device-policy` | 是 | 策略代表设备当前应执行的规则，新上线设备必须立即获取 |
| 语音补盲命令 | `smart-cat/lubancat3-demo-001/voice-commands` | 否 | 语音命令是即时动作，不应在设备重连后重复执行旧命令 |

因此，普通消息更适合“发生过的一次性事件”，retained 消息更适合“当前状态或当前配置”。本项目用 retained 消息保存运行参数和设备策略，正好体现了云边协同中“平台改变设备行为，设备上线后自动获取最新状态”的设计思想。

## 语音补盲与人工确认

为了避免系统在未知商品上“装作识别正确”，项目设计了语音补盲和人工确认机制。低置信度或未知商品进入待确认状态后，操作者可以在 Web 后台修改商品类别、单价并重新计算总价，也可以通过手机访问 Web 页面，用语音或文本输入“这是苹果”“改成番茄”等命令完成修正。

修正后系统会记录修正前后的商品、单价、总价和操作来源，并通过 MQTT 下发 `speak_text` 命令，让板端 SYN6288 播报修正后的报价。这一设计体现了“AI 自动识别 + 人工兜底确认”的可靠业务流程。

除商品修正外，语音补盲还承担无障碍查询和现场运维辅助作用。用户或店员不需要一直盯着屏幕，也可以通过简单口令获取设备状态、重量、价格和最近交易信息。

| 场景 | 用户操作 | 系统响应 |
| --- | --- | --- |
| 视障顾客 | 说“重量” | TTS 念“当前秤上重量约 200 克” |
| 拿手菜的顾客 | 说“价格” | TTS 念“苹果，单价 5 元一斤，总价 2 块” |
| 看不清屏幕 | 说“最近交易” | TTS 念“最近一笔交易，商品苹果，状态已可靠识别……” |
| 店员 / 运维 | 说“状态” | TTS 念“设备当前空闲待机……” |

这部分让语音模块不只是“报价喇叭”，而是成为边缘设备的人机交互入口。对于演示来说，它可以体现项目对视障顾客、嘈杂环境和无屏操作场景的考虑。

## 项目完成度与展示重点

当前项目已经具备完整演示链路：

```text
硬件连线完整：摄像头、HX711、SYN6288、无线网络
边缘端可常驻运行：放上商品自动触发交易
AI 推理已部署到 NPU：YOLO11 -> ONNX -> RKNN
业务闭环完整：识别、称重、计价、播报、记录、上报
Web 后台可用：交易、商品、设备、统计、策略和语音补盲
云边协同可展示：后台配置下发，板端接收并执行，事件回传
异常处理明确：低置信度、未知商品进入待确认和补盲流程
```

答辩时可以重点强调三点：

```text
1. 这不是单点算法演示，而是一个端侧设备和后台平台协同运行的系统。
2. 选择 RKNN/NPU 是为了让模型真正适配 RK3576 边缘硬件，提高实时性。
3. 项目保留人工确认和语音补盲，解决真实场景中未知商品和低置信度识别的问题。
```

最终成品可以概括为：一套面向生鲜称重收银场景的边缘智能识别终端原型，具备本地 AI 推理、传感器融合、动态语音播报、远程参数/策略下发、交易数据同步和后台管理能力。

## 后续改进方向

当前项目为了便于调试、迁移和演示，交易记录、设备事件、策略历史等数据主要采用 JSONL 文件保存。JSONL 的优点是结构简单、追加写入方便、直接打开即可查看，也适合项目原型阶段快速排查问题。但随着交易量增加，后续可以将数据层升级为 SQLite：

```text
交易流水 -> transactions 表
设备事件 -> device_events 表
策略历史 -> policy_history 表
语音命令 -> voice_commands 表
商品价格历史 -> product_price_history 表
```

SQLite 能提供更好的索引查询、分页筛选、条件统计和数据一致性，适合项目从演示原型继续向长期运行系统演进。当前阶段保留 JSONL，可以降低改动风险，保证硬件闭环和 Web 平台稳定交付。
